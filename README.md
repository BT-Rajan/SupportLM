# SupportLM

A self-hosted, multi-tenant AI customer support assistant. A company uploads
its own documents (or syncs them from its website), visitors ask questions in
an embeddable chat widget, and answers are generated by an LLM with citations
back to source content — with a full admin console for content review,
escalation to human support, and usage analytics.

Originally built as a single-tenant proof of concept ("KnowledgeLM"); the
current architecture is multi-tenant from the ground up.

## Status

All 8 planned phases are complete except Phase 8 (in progress). See
`docs/STATUS.md` for the authoritative, round-by-round build log, and
`docs/MASTER_PROMPT.md` for the full scope this was built against.

## Stack

- **Backend:** Python 3.11+, FastAPI
- **DB:** MySQL/MariaDB, shared schema with `tenant_id` row-level scoping
  (no schema-per-tenant, no ORM — raw SQL via a thin connection pool)
- **Retrieval:** hybrid search — MySQL FULLTEXT keyword search fused with
  brute-force cosine similarity over embeddings stored as JSON (no vector DB)
- **Chat LLM:** pluggable per tenant — DeepSeek, OpenAI, or Anthropic
- **Embeddings:** always local (`sentence-transformers`), regardless of which
  chat provider a tenant picks — no embeddings provider has a public API
  across all three, and switching models mid-flight would invalidate every
  stored vector
- **Frontend:** thin server-rendered Jinja2 shells (`templates/chat.html`,
  `templates/admin.html`) mounting Vue 3 single-page apps built from
  `frontend/src/` (two separate Vite entry points, one per surface,
  sharing one `tokens.css`) — compiled to `static/dist/`, not written
  or edited by hand

## Project layout

```
  core/         config, security/session, RBAC, tenant resolution, LLM

                escalation, analytics, etc.

                tenant/admin, set branding, change tenant status)

                admin-widget.{js,css}) — generated by `npm run build`,

                compiled Vue bundles above (embeddable widget + console)

                is the shared design-token source of truth, see
                docs/DESIGN_SYSTEM.md
docs/           MASTER_PROMPT.md (scope), STATUS.md (build log),
                Phase N WBS.md (per-phase work breakdown),
                DESIGN_SYSTEM.md (UI tokens/patterns)
tests/          pytest — DB-dependent tests skip cleanly without one
                configured (see Setup below)
```

## API surface, by function

Every tenant-scoped route lives under `/t/{tenant_slug}/...`.

| Area | Prefix | Router |
|---|---|---|
| Chat widget (ask, transcript email, feedback, escalation) | `/api/chat` | `app/api/chat.py` |
| Documents (upload, review workflow, website sync, duplicate scan) | `/api/documents` | `app/api/documents.py` |
| Categories | `/api/categories` | `app/api/categories.py` |
| Admin auth (login/logout/logout-all) | `/api/auth` | `app/api/auth.py` |
| API keys (programmatic access) | `/api/api-keys` | `app/api/api_keys.py` |
| LLM provider config (per tenant) | `/api/tenant/llm-config` | `app/api/llm_config.py` |
| System prompt versions (edit/rollback) | `/api/tenant/prompt-versions` | `app/api/prompt_versions.py` |
| Support inbox config (escalation emails) | `/api/tenant/support-config` | `app/api/support_config.py` |
| Usage analytics | `/api/tenant/analytics` | `app/api/analytics.py` |
| Audit log | `/api/tenant/audit-log` | `app/api/audit.py` |

Two non-API routes: `/t/{tenant_slug}/` (chat widget) and
`/t/{tenant_slug}/admin` (admin console).

## Setup

```bash
cp .env.sample .env              # fill in DB creds, at least one LLM key
pip install -r requirements.txt
python scripts/init_db.py        # creates the DB, runs every migration in order
python scripts/create_tenant.py "My Company" my-company --owner-email owner@example.com --owner-password yourpassword
uvicorn app.main:app --reload
```

Then visit `/t/my-company/` for the chat widget and `/t/my-company/admin` to
log in and manage content.

`static/dist/` (the compiled chat widget and admin console) is committed,
so the command above works with no Node toolchain involved. Only rebuild
it if you're changing something under `frontend/src/`:

```bash
cd frontend
npm install
npm run build                    # writes static/dist/{chat,admin}-widget.{js,css}
```

`scripts/create_admin.py`, `scripts/set_tenant_branding.py`, and
`scripts/set_tenant_status.py` cover the remaining one-off operator tasks
that don't have an admin-UI equivalent yet (see each script's own docstring).

## Feature areas, by phase

- **Multi-tenancy & org foundation** — shared-schema tenant isolation,
  per-tenant branding, usage tiers/plan limits (`document`, `tenant`,
  `tenant_user`, `plan_tier`, `tenant_branding`).
- **Access control & anonymous chat** — RBAC (owner/admin/editor/viewer),
  API keys, session hardening (server-side revocation, logout-everywhere),
  opt-in transcript email for anonymous chat visitors.
- **Knowledge base management** — draft → review → publish workflow for
  documents (nothing is instant-live), manual website content sync
  (hash-diffed against existing docs), duplicate/near-duplicate title and
  heading detection.
- **Retrieval & answer quality** — hybrid (keyword + semantic) search,
  pluggable per-tenant LLM provider (DeepSeek/OpenAI/Anthropic), system
  prompt versioning with rollback.
- **Conversation experience** — multi-turn memory (follow-ups use prior
  turns as context), multi-language support, thumbs up/down feedback.
- **Escalation to service request** — when the assistant can't help,
  generates a per-tenant-sequential SR number, emails both the tenant's
  support inbox and the visitor with the full transcript.
- **Analytics & reporting** — usage dashboard, flagged/low-confidence
  question log, CSAT from feedback, per-tenant LLM cost tracking, CSV export.
- **Admin, ops & webhooks** *(in progress)* — audit log (done); rate
  limiting, agent/bot configuration UI, health/status page, and webhooks
  are not yet built.

## Key design decisions

- **Shared schema, not schema-per-tenant.** Every tenant-scoped table has a
  `tenant_id` column and a matching index; isolation is enforced in the
  query layer (`app/core/tenant_scope.py`, `app/core/rbac.py`), not by
  separate databases. Simpler migrations and operations at this project's
  scale; revisit only if a specific tenant needs hard isolation for
  compliance reasons.
- **MySQL for everything, including embeddings — no vector DB.** Embeddings
  are stored as JSON float arrays and compared with brute-force cosine
  similarity in Python, fused with native MySQL FULLTEXT search
  (`app/services/vector_store.py`). Fine at the scale this is built for;
  the retrieval interface is isolated behind a small protocol so a real
  vector store could be swapped in later without touching callers.
- **No ORM.** Raw SQL via a thin connection pool wrapper
  (`app/db/pool.py`), typed dataclasses for results. Keeps queries and
  the schema explicit and auditable.
- **Embeddings always run locally**, independent of whichever chat LLM
  provider a tenant has configured — see Stack, above.
- **Manual triggers, not background jobs.** Website sync and duplicate
  scanning are both admin-initiated ("Sync now" / "Scan for duplicates"),
  not cron-scheduled — this project deliberately has no job queue.

## Running the tests

```bash
python -m pytest tests/ -q
```

Tests that need a real database skip cleanly (not fail) if one isn't
configured. To run the full suite, point `.env` at a real MySQL/MariaDB
instance and run `python scripts/init_db.py` first.
