# 5.1 — Usage Tier Structure

Status: **confirmed — implemented in `migrations/007_plan_tiers.sql`**

## Confirmed tiers

| Tier       | Doc limit | Messages/month | Seats     |
|------------|-----------|-----------------|-----------|
| Starter    | 25        | 500             | 2         |
| Pro        | 200       | 5,000           | 10        |
| Enterprise | Unlimited | Unlimited       | Unlimited |

`NULL` in `plan_tier` = unlimited (used for Enterprise on all three
columns).

## Enforcement behavior (confirmed for 5.2)

- **Documents:** hard block. Upload is rejected with a 403 once the
  tenant is already at `doc_limit`.
- **Messages:** soft warn. Chat keeps answering; the response carries a
  `limit_warning` string once the tenant has reached `message_limit`
  for the current calendar month. Nothing is rejected.
- **Seats:** limit is stored (`seat_limit`) but not enforced anywhere
  yet — no route creates additional `tenant_user` rows beyond the
  single owner link `create_tenant.py` makes. Phase 2 (RBAC / user
  management) is where seat enforcement gets an actual call site.

## Migration notes

`tenant.plan_tier` existed since `002_tenant_org.sql` as an FK-less
placeholder (`VARCHAR(50) DEFAULT 'free'`). `007_plan_tiers.sql`:
1. Creates `plan_tier` and seeds the three rows above.
2. Remaps any existing `plan_tier = 'free'` tenants to `'starter'` —
   `'free'` was never a real tier name, just a placeholder default.
3. Adds the FK `tenant.plan_tier -> plan_tier.slug` and changes the
   column default to `'starter'`.

`scripts/create_tenant.py` now inserts new tenants with `'starter'`
instead of the old `'free'` literal.
