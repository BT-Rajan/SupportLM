# 1.1 — Tenant/Org Schema Proposal

Status: **confirmed — implemented in `migrations/002_tenant_org.sql`**

## Decision carried over from `STATUS.md`
Shared MySQL schema, `tenant_id` row-level scoping (not schema-per-tenant).

## New tables

### `tenant`
```
id          INT PK AUTO_INCREMENT
name        VARCHAR(255) NOT NULL
slug        VARCHAR(150) NOT NULL UNIQUE   -- subdomain / URL-safe identifier
plan_tier   VARCHAR(50)  NOT NULL DEFAULT 'free'   -- FK-less for now; 5.1 defines the real tier table
status      ENUM('active','suspended','trial') NOT NULL DEFAULT 'trial'
created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
updated_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
```

### `tenant_user`
Exists now so it's assignable from Phase 1, even though RBAC (roles beyond
a boolean-ish owner/admin) is built out in Phase 2.
```
id          INT PK AUTO_INCREMENT
tenant_id   INT NOT NULL, FK -> tenant(id) ON DELETE CASCADE
admin_id    INT NOT NULL, FK -> admin_user(id) ON DELETE CASCADE
role        ENUM('owner','admin') NOT NULL DEFAULT 'admin'   -- placeholder; Phase 2 RBAC extends this
created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
UNIQUE (tenant_id, admin_id)
```

## `company` vs `tenant`

Recommendation: **fold `company` into `tenant`.** `company` today is a
single-row settings-like table (name + free-form `profile_json`) — that's
just a tenant's profile. Proposal:

- Add `profile_json JSON NULL` to `tenant` directly.
- Drop the standalone `company` table; migration 1.2 moves any existing
  `company` row's data into the corresponding `tenant` row's `profile_json`
  during backfill (1.3), then drops `company`.
- Anywhere `company_id` might have been implicitly assumed (there currently
  isn't a `company_id` column on other tables — `company` was a singleton),
  no FK rewiring needed beyond what 1.2 already does for `tenant_id`.

This avoids carrying two parallel "who does this belong to" concepts into
Phase 2+.

## Tables getting `tenant_id` (→ 1.2)
`document`, `document_chunk`, `embedding`, `category`, `conversation`,
`message`, `citation`, `agent`.

Note: `document_chunk`, `embedding`, `message`, `citation` are all reachable
via a join back to a `tenant_id`-bearing parent (`document` or
`conversation`) already. Proposal is to **still add `tenant_id` directly to
each** (denormalized) rather than rely on joins, because:
- 3.1's middleware needs a single flat filter it can apply to every query,
  not a per-table join strategy.
- 1.4's indexes are simpler and faster as flat `(tenant_id, ...)` composites.
- 3.3's cross-tenant tests are simpler to assert against a flat column.

`admin_user` does **not** get a `tenant_id` column — an admin can belong to
multiple tenants via `tenant_user`, which is the whole reason that join
table exists.
