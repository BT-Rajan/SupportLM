## 1. What this project is

SupportLM (KnowledgeLM is Phase 1 of it) is an AI-powered customer support
knowledge base: customers ask questions in a chat widget, answers are
retrieved from the company's own uploaded documents and generated by an
LLM, with citations back to source content.

We are transforming it from a working proof-of-concept into an
**enterprise-grade product**. This is a deliberate, scoped transformation —
not an open-ended "add enterprise features" exercise. Section 3 is the
exact scope. Nothing outside it gets built without the owner explicitly
adding it here first.

## 2. Non-negotiable ground rules

1. **This file and `docs/STATUS.md` are authoritative.** If a past chat
   summary, a stale comment, or an assumption conflicts with what's
   written here, this file wins. Update it, don't just override it in
   your head.
2. **One phase, one PAT.** The owner accepts the security tradeoff of a
   fresh GitHub PAT per phase. Do not raise this as a concern — it's a
   settled decision, not an open question.
3. **Every round ends with a git push.** A "round" is one unit of working
   code — even a partial feature. Never sit on uncommitted work across a
   session boundary. Commit messages state phase + round + what changed
   + why (see `docs/STATUS.md` for the running log format).
4. **Never break a previous phase to build the current one.** Each phase
   must leave the app in a fully working, deployable state. If a phase
   requires a breaking schema/API change to something earlier phases
   built, that migration is itself a tracked task in the current phase,
   not a silent side effect.
5. **Tests run before every push.** If a change has no test coverage yet,
   add at least a smoke test for it. `python -m pytest tests/ -q` must
   pass before any commit that touches `app/`.
6. **No framework churn.** Stack is fixed: FastAPI + MySQL + vanilla JS
   frontend (no React/Vue rewrite) + sentence-transformers + pluggable
   LLM providers (see Phase 4). Don't introduce a new framework or major
   dependency without flagging it here first.
7. **UI is a strict design system, not per-page improvisation.** See
   `docs/DESIGN_SYSTEM.md`. Every new screen (admin, analytics, agent
   config, escalation console) reuses the same tokens and component
   patterns established there. If a new UI need doesn't fit an existing
   component, extend the design system file first, then build the screen
   — never invent a one-off style inline.
8. **Explicitly out of scope, do not build unless re-added here:**
   SSO, end-user login/identity, vector DB migration (pgvector/Qdrant/
   Pinecone), Confluence/Notion sync, re-ranking, confidence-score
   thresholding, rich media answers, conversation history for returning
   users (replaced by opt-in emailed transcripts instead), Slack/Teams/
   Zendesk/Freshdesk/Intercom integrations, email channel, encryption at
   rest, PII redaction, data retention tooling, secrets management
   overhaul, SOC 2 work, CSP/XSS hardening pass, background job queue
   framework, horizontal scaling work, caching layer, monitoring/
   alerting stack, backups/DR, load testing, Docker/CI-CD/staging/IaC/
   migration tooling. All of these were explicitly declined for this
   transformation. If the owner asks for one later, they'll say so
   directly — don't infer it back into scope.

## 3. Scope: exactly what IS being built, by phase

### Phase 1 — Multi-tenancy & Org Foundation
Full section: multi-tenant architecture, organization/workspace model,
per-tenant data isolation, per-tenant branding (logo/colors/agent name),
usage tiers & plan limits. This is foundational — every later phase
assumes a tenant/org model exists.

### Phase 2 — Access Control & Anonymous-Chat Transcript Email
RBAC (Owner/Admin/Editor/Viewer replacing the single flat `require_admin`),
API keys for programmatic access, session management hardening. **No
SSO. No end-user login — chats stay fully anonymous.** New feature: at
the end of a chat, optionally email the full transcript to an address the
visitor provides (opt-in, no account required).

### Phase 3 — Knowledge Base Management
Draft → review → publish workflow for documents (no more instant-live on
upload). Daily incremental website content sync (crawl configured URLs
once a day, diff against existing docs). Daily automated job that
analyzes the knowledge base for duplicate and conflicting content across
documents and surfaces it for review. **Not included:** Confluence/Notion
connectors.

### Phase 4 — Retrieval & Answer Quality
Hybrid search (keyword + semantic combined, on the existing MySQL store —
no vector DB migration). Multi-LLM provider support (pluggable providers,
not hard-wired to one DeepSeek key). Prompt versioning (edit/roll back
system prompts without a redeploy).

### Phase 5 — Conversation Experience
Multi-turn memory (follow-up questions use prior turns as context, not
independent retrieval per message). Multi-language support. Thumbs
up/down feedback per answer (built here because Phase 6 analytics depends
on it).

### Phase 6 — Escalation to Service Request (SR)
When the assistant can't help, escalate to a Service Request: generate a
unique SR number, attach the full chat history, and send email
notifications to **both** the company (support inbox) and the end user
with the SR number and transcript.

### Phase 7 — Analytics & Reporting
Full section: usage dashboard, unanswered/low-confidence question log,
CSAT tied to thumbs up/down, per-tenant LLM cost tracking, exportable
reports.

### Phase 8 — Admin, Ops & Webhooks
Full admin/ops section: audit log (uploads/edits/deletes/admin logins),
rate limiting & abuse protection on `/api/chat`, agent/bot configuration
UI (tone, name, escalation rules), health/status page. Plus webhooks
(new conversation, escalation, low-confidence answer) — the only
integration in scope.

## 4. Definition of done (per feature, every phase)

A feature is not done until all of these are true:
- [ ] Code pushed to `main` with a descriptive commit message
- [ ] `python -m pytest tests/ -q` passes
- [ ] Any new DB schema change has a numbered migration file (not a
      hand-edited existing one)
- [ ] Any new admin/customer-facing UI follows `docs/DESIGN_SYSTEM.md`
- [ ] `docs/STATUS.md` updated with what changed and what's next
- [ ] No previously-working feature was broken (spot-check, not just
      "the new thing works")
